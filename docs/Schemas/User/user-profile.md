# User Schema

A **User** represents a person registered in the system.
It includes identity info, profile data, media, social links, and account state.

---

## User Object (High-Level)

```ts
User {
  username: string
  phone: string
  email?: string
  userState: "online" | "offline"
  avatar: Media | null
  coverImage: Media | null
  status: string
  dateOfBirth: string (ISO Date)
  bio: string | null
  location: string | null
  gender: "male" | "female" | "other" | "not specify"
  socialLinks: SocialLink[]
  visibility: boolean
  accountStatus: "active" | "inactive" | "suspended"
  createdAt: string
  updatedAt: string
}
```

---

## Field-by-Field Explanation

### 1. Identity & Core Info

#### `username: string`

Unique display name of the user.

- Auto-generated by default
- Used for search, mentions, profile URLs

Example:

```json
"username": "user_8f3a2d9"
```

---

#### `phone: string`

User’s phone number (primary identifier).

Example:

```json
"phone": "+923001234567"
```

---

#### `email?: string`

Optional email address.

- Always lowercase
- May be `null` or missing

Example:

```json
"email": "adeel@gmail.com"
```

---

### 2. Online Presence

#### `userState: "online" | "offline"`

Represents real-time presence.

Used for:

- chat apps
- status indicators (green dot etc.)

Example:

```json
"userState": "online"
```

---

### 3. Profile Media

Both `avatar` and `coverImage` share the same structure.

### Media Object

```ts
Media {
  url: string
  publicId: string
}
```

#### `avatar: Media | null`

User profile picture.

Example:

```json
"avatar": {
  "url": "https://cdn.app.com/avatar123.jpg",
  "publicId": "avatars/user_123"
}
```

---

#### `coverImage: Media | null`

Profile cover / banner image.

Same structure as avatar.

---

### 4. Profile Info

#### `status: string`

Short one-liner about the user.

Think:

> WhatsApp bio / Discord status

Example:

```json
"status": "hi i am available"
```

---

#### `bio: string | null`

Longer description.

Example:

```json
"bio": "Full stack dev. Coffee addict."
```

---

#### `location: string | null`

User’s location.

Example:

```json
"location": "Karachi, Pakistan"
```

---

#### `dateOfBirth: string`

ISO date string.

Example:

```json
"dateOfBirth": "2005-08-14T00:00:00.000Z"
```

---

#### `gender`

Enum value:

- `"male"`
- `"female"`
- `"other"`
- `"not specify"`

Example:

```json
"gender": "male"
```

---

### 5. Social Links

#### `socialLinks: SocialLink[]`

Array of external profile links.

### SocialLink Object

```ts
SocialLink {
  platform: string
  link: string
}
```

Example:

```json
"socialLinks": [
  {
    "platform": "github",
    "link": "https://github.com/suffa"
  },
  {
    "platform": "tiktok",
    "link": "https://t.me.com/suffa"
  }
]
```

Used for:

- profile pages
- link-in-bio features
- social icons

---

### 6. Privacy & Visibility

#### `visibility: boolean`

Controls whether profile is publicly visible.

- `true` → public
- `false` → hidden / private

Example:

```json
"visibility": true
```

---

### 7. Account State

#### `accountStatus`

Enum:

- `"active"` → normal user
- `"inactive"` → deactivated
- `"suspended"` → banned / restricted

Frontend use cases:

- block login
- show warning banners
- disable actions

Example:

```json
"accountStatus": "active"
```

---

### 8. System Fields

These are auto-managed by the system.

#### `createdAt`

When user was created.

#### `updatedAt`

Last profile update.

Example:

```json
"createdAt": "2026-01-10T12:00:00.000Z",
"updatedAt": "2026-02-03T18:20:00.000Z"
```

---

## Things Frontend Should Know (Important)

### 1. Some fields may be hidden

Certain endpoints may not return:

- `socialLinks`
- `blocked users`
- internal flags

Depends on:

- authentication
- permissions
- role

---

### 2. Searchable Fields

The backend allows searching users by:

- username
- bio
- location

So you can build:

- global search
- user discovery
- suggestions

---

### 3. Null is normal

These fields can be `null`:

- avatar
- coverImage
- bio
- location
- email

Always handle safely in UI.

---

## Mental Model (Frontend POV)

Think of the User object in 4 layers:

```
1. Identity
   → username, phone, email

2. Presence
   → userState, status

3. Profile
   → avatar, coverImage, bio, gender, location

4. System
   → visibility, accountStatus, timestamps
```
